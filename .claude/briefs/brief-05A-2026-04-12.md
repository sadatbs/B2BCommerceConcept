# Brief #05A — Domain Model Correction: Cart Ownership & Requisition Workflow
**Date:** 2026-04-12  
**Tag:** v0.5.1  
**Branch:** feature/05A-domain-correction-requisition-workflow

---

## Scope

Corrected a domain model inaccuracy introduced in Brief #05: Cart was owned by a Customer (company) when it should belong to an individual User (buyer). Introduced a full Requisition workflow to replace the removed direct `POST /api/orders` endpoint, giving B2B procurement a proper approval gate.

---

## What Was Done

### Domain Layer
- **`Cart`**: Replaced `Guid? CustomerId` (nullable, company-owned) with `Guid UserId` (required, user-owned) + `Guid CustomerId` (required, company context). `Cart.Create(Guid userId, Guid customerId)` now validates both are non-empty.
- **`User`**: Added `decimal? BudgetLimit` property with `SetBudgetLimit(decimal? limit)` mutation method. `User.Create(...)` signature extended with optional `budgetLimit` parameter.
- **`Requisition`** (new): AggregateRoot with `CreateFromCart(Cart, productSnapshot)` that freezes prices at submission time. State machine: `Submitted → Approved/Rejected → Ordered` via `Approve()`, `Reject(reason)`, `MarkOrdered()`.
- **`RequisitionLineItem`** (new): Snapshot entity — `Sku`, `ProductName`, `UnitPrice` frozen at creation. No FK to Products table by design (price snapshot independence).
- **`RequisitionStatus`** (new): `Submitted = 0`, `Approved = 1`, `Rejected = 2`, `Ordered = 3`.
- **`Order`**: Added `Guid? RequisitionId` for traceability. Added `CreateFromRequisition(Requisition)` factory — requires `Status == Approved`, copies frozen line items.
- **`OrderItem`**: Added `CreateFromRequisitionLineItem(Guid orderId, RequisitionLineItem)` factory.
- **`ICartRepository`**: Renamed `GetByCustomerIdAsync` → `GetByUserIdAsync`.
- **`IRequisitionRepository`** (new): Full paged query interface.

### Infrastructure Layer
- **`CartConfiguration`**: `UserId` required + unique index (one active cart per user); `CustomerId` required, non-nullable.
- **`RequisitionConfiguration`** (new): `Status` stored as `string`, navigation via field access mode, indexes on `UserId`, `CustomerId`, `Status`, `SubmittedAt`.
- **`RequisitionLineItemConfiguration`** (new): Snapshot design — no FK to Products.
- **`OrderConfiguration`**: Added `RequisitionId` nullable column + index.
- **`UserConfiguration`**: Added `BudgetLimit` with `HasPrecision(18, 2)`.
- **`CommerceDbContext`**: Added `DbSet<Requisition>`, `DbSet<RequisitionLineItem>`, applied new configurations.
- **`CartRepository`**: Renamed method to query on `c.UserId`.
- **`RequisitionRepository`** (new): Standard EF implementation.
- **Migration**: `AddRequisitionWorkflowAndCartOwnership` — alters `Carts`, extends `Users` and `Orders`, creates `Requisitions` and `RequisitionLineItems` tables.

### Contracts Layer
- **`CreateCartRequest`**: Now `required Guid UserId` + `required Guid CustomerId` (both mandatory).
- **`CartDto`**: Added `required Guid UserId`; `CustomerId` from nullable to required.
- **`CreateUserRequest`**: Added `decimal? BudgetLimit`.
- **`UserDto`**: Added `decimal? BudgetLimit`.
- **`Requisitions/`** (new — 5 files): `RequisitionDto`, `RequisitionDetailDto`, `RequisitionLineItemDto`, `SubmitRequisitionRequest`, `RejectRequisitionRequest`.

### API Layer
- **`CartEndpoints`**: `CreateCart` uses `CreateCartRequest.UserId` + `CustomerId`.
- **`CustomerEndpoints`**: `CreateUser` passes `request.BudgetLimit` to `User.Create`.
- **`OrderEndpoints`**: Removed `POST /` (direct order creation). Orders are now exclusively created via requisition approval.
- **`RequisitionEndpoints`** (new, 5 endpoints):
  - `POST /api/requisitions` — SubmitRequisition: load cart → verify ownership → build price snapshot → create Requisition → budget check → auto-approve (no BudgetLimit or total ≤ limit) or await approval
  - `GET /api/requisitions/{id}` — GetRequisitionById with line items
  - `GET /api/requisitions/user/{userId}` — paginated list for buyer
  - `PATCH /api/requisitions/{id}/approve` — role gate (`requestingUserRole=Approver`); creates Order + marks Ordered
  - `PATCH /api/requisitions/{id}/reject` — role gate; stores rejection reason
- **`Program.cs`**: Registered `IRequisitionRepository`, `ConfigureWarnings` to suppress `PendingModelChangesWarning` (EF Core 9 false positive — `ef migrations has-pending-model-changes` confirms no pending changes).

### Test Layer
- **`CartTests.cs`**: All 9 tests updated: `Cart.Create()` → `Cart.Create(Guid.NewGuid(), Guid.NewGuid())`.
- **`OrderTests.cs`**: Helper + 3 direct calls updated to new `Cart.Create` signature (`CreateFromCart` kept for backward compatibility).
- **`CartEndpointsTests.cs`**: All `new CreateCartRequest()` → `new CreateCartRequest { UserId = ..., CustomerId = ... }`.
- **`CartPricingTests.cs`**: All `CreateCartRequest` usages updated; `Cart_WithoutCustomer_ShowsListPrice` now uses a non-existent Guid for CustomerId (pricing service falls back to list price).
- **`OrderEndpointsTests.cs`**: Removed 3 `CreateOrder_*` tests (endpoint gone); `CreateOrderAsync` helper rewritten to go through full requisition workflow (Customer → User (no budget limit) → Cart → Submit → auto-approve → fetch from orders list).
- **`RequisitionTests.cs`** (new — 7 unit tests): `CreateFromCart` price freeze, empty cart, `Approve`, double-approve throws, `Reject`, empty-reason throws, `MarkOrdered`.
- **`RequisitionEndpointsTests.cs`** (new — 8 integration tests): auto-approve path, requires-approval path, empty cart 400, ownership mismatch 400, get by ID, approve by Approver, approve by non-Approver (403), reject by Approver.

## Decisions & Rationale

| Decision | Rationale |
|---|---|
| `Cart.UserId` required (not optional) | Every cart must have an owner in the B2B model; anonymous carts are not a use case |
| `CustomerId` also required on Cart | Company context needed for pricing tier lookup; never makes sense without it |
| Price snapshot at requisition creation | Prices can change between submission and approval; the buyer's submitted total must be immutable for audit |
| `RequisitionLineItem` has no FK to Products | Snapshot independence — products can be deleted/modified without corrupting historical requisitions |
| Auto-approve when `BudgetLimit` is null | Null = no limit set = unlimited authority = auto-approve is the safe default |
| `TypedResults.StatusCode(403)` over `TypedResults.Forbid()` | `ForbidHttpResult` invokes auth challenge middleware; without an auth scheme configured it produces 400. `StatusCode(403)` directly sets the response code without any middleware delegation |
| `PendingModelChangesWarning` suppressed | EF Core 9.0.4 runtime validator and `has-pending-model-changes` CLI use different algorithms; CLI confirms no pending changes. This is a known false positive in EF Core 9. Suppressed via `ConfigureWarnings` |
| Kept `Order.CreateFromCart` | Backward-compatible; unit tests for Order state machine still use it via the helper; removing it would be scope creep |
