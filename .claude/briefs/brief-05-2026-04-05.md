# Brief #05 — Customer, User & Contract Pricing

**Date:** 2026-04-05
**Branch:** `feature/05-customer-pricing`
**Tag:** `v0.5.0`

---

## Scope

Introduce B2B customer management (companies and their users), price tiers with per-product override prices, a domain pricing service, and wire the pricing service into cart line-item price resolution.

---

## What Was Done

### Domain Layer
- `PaymentTerms` enum (DueOnReceipt, Net15, Net30, Net45, Net60, Net90)
- `UserRole` enum (Buyer, Admin, Approver)
- `Customer` entity — code (normalized to uppercase), email (normalized to lowercase), `PaymentTerms`, optional `PriceTierId` FK, `List<User>` navigation, `Activate`/`Deactivate`/`AssignPriceTier`/`UpdateDetails` methods
- `User` entity — belongs to `Customer`, email normalized to lowercase, `FullName` computed property, `UpdateDetails`/`Activate`/`Deactivate` methods
- `PriceTier` entity — name (trimmed), optional description, `List<TierPrice>` navigation, `UpdateDetails` method
- `TierPrice` entity — composite key `(PriceTierId, ProductId)`, `UpdatePrice` method, `UpdatedAt` tracking
- `IPricingService` — `GetPriceAsync` / `GetPricesAsync` for single and batch price resolution
- Repository interfaces: `ICustomerRepository`, `IUserRepository`, `IPriceTierRepository`

### Infrastructure Layer
- EF configurations for all four entities using `PropertyAccessMode.Field` for list navigations and `HasConversion<string>()` for `PaymentTerms`
- `TierPriceConfiguration` uses composite PK `{ PriceTierId, ProductId }`
- `CustomerConfiguration` sets `OnDelete(DeleteBehavior.SetNull)` for the `PriceTier` FK
- Repository implementations with eager loading where required
- `PricingService` — batch-loads list prices and tier prices, merges with fallback: tier price if available, otherwise list price
- Migration: `AddCustomerUserPricing` (2026-04-05)

### API Layer
- `CustomerEndpoints` (10 endpoints): CRUD, activate/deactivate, assign price tier, list/create users
- `PriceTierEndpoints` (8 endpoints): CRUD, set/remove/list tier prices
- `CartMappingExtensions` updated with async `ToDtoAsync` overload injecting `IPricingService`; `CartEndpoints` updated to use it
- Validators: `CreateCustomerRequestValidator`, `UpdateCustomerRequestValidator`, `CreateUserRequestValidator`, `CreatePriceTierRequestValidator`, `SetTierPriceRequestValidator`
- Mapping extensions: `CustomerMappingExtensions`, `UserMappingExtensions`, `PriceTierMappingExtensions`
- `Program.cs` — registered `ICustomerRepository`, `IUserRepository`, `IPriceTierRepository`, `IPricingService`

### Tests
- Domain unit tests: `CustomerTests` (9), `UserTests` (8), `PriceTierTests` / `TierPriceTests` (11 combined)
- Integration tests: `CustomerEndpointsTests` (14), `PriceTierEndpointsTests` (10), `CartPricingTests` (4)

## Decisions

**`PricingService` batch approach:** `GetPricesAsync` fetches all needed list prices and tier prices in two queries rather than N+1 per line item. The fallback chain is: tier price → list price.

**`CartMappingExtensions` async overload:** Rather than making the entire mapping synchronous (requiring a price lookup per item), introduced `ToDtoAsync` that takes a pre-resolved price dictionary. The old synchronous `ToDto` is kept for empty-cart fast paths.

**`GetCustomerUsers` handler parameter ordering:** ASP.NET Core minimal API returns an empty 400 when `[AsParameters]` is used alongside two DI-injected repository parameters in the same handler (the root cause is unknown — `GetCatalogProducts` with one repo works fine). Workaround: reorder so DI services come before optional query params (`int? page = null, int? pageSize = null`). This is valid C# (optional params must follow required params) and minimal API resolves each parameter from the correct source.

**`TierPrice` composite PK:** Used explicit `HasKey(tp => new { tp.PriceTierId, tp.ProductId })` rather than a surrogate Guid. The composite key accurately models the business constraint (one price per product per tier) and avoids a redundant Id column.

**`HasConversion<string>()` for `PaymentTerms`:** Stored as string for readability in the database; the enum's integer value carries no external meaning.
